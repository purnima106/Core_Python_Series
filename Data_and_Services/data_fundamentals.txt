#Data related notes
dimensions holds all the contextual data/descriptive information or data
for eg cust_name,prod_name,etc but these values changes overtime.
these changes are handled by Slowly Changing dimensions

#Type 1 SCD (Overwrite / Upsert)-Just overwrite the old data with the new data (no history)
Upsert = Update + Insert

Real-world Examples
Fixing spelling mistakes in names
Updating incorrect email
Latest product price (if history not needed)

When NOT to Use It

Don’t use Type 1 when:

❌ You need history
❌ You want to track changes over time

#Type 2 and 3- “History” & “Previous Value” in SCD
*History= Full change tracking/all past versions of a record are stored

Example:Customer lived in Mumbai before
Then moved to Pune
-This is used in SCD Type 2

Previous Value (Only Last Value)

*Previous value = you keep just one old value (not full history)
Use Type 3:
You need only last change
You know:

Current = Pune
Just the last value = Mumbai

❌ But you don’t know anything older than that

A data warehouse = clean, organized, ready-to-use data

Characteristics:
Data is structured (tables, rows, columns)
Already cleaned & transformed
Fast for analytics (SQL queries)

Real tools:
Snowflake
Amazon Redshift
Google BigQuery

A data lake = dump everything as-is

Characteristics:
Stores raw data
Any format:
JSON
CSV
images
logs
Not cleaned yet

Real tools:
Amazon S3
Azure Data Lake

A lakehouse = data lake + warehouse features

Think of it like a smart home 🏡:

Raw storage (like lake)
Organized + fast queries (like warehouse)

✅ Characteristics:
Stores raw + structured data
Supports SQL queries
Adds schema + performance on top of lake

🛠️ Real tools:
Databricks
Delta Lake

Real Pipeline Example
App Data → Data Lake (S3) → Processing → Warehouse (Snowflake)
                       ↓
                  Lakehouse (Databricks)

File Formats (Row based and Column based)

Feature	        Row-Based 🔴	Column-Based 🟢
Storage style	Row-wise	Column-wise
Best for	    Transactions	Analytics
Query type	    SELECT *	SELECT few cols
Speed (analytics)	❌ Slower	✅ Faster
Compression	    ❌ Less	✅ High

*Parquet is a

column-based file format designed for fast analytics

It’s mainly used in:

Data lakes (like S3)
Big data systems
Data warehouses

Where You See It in Real World:

-Amazon S3 → store Parquet files
-AWS Glue → convert raw data → Parquet
-Amazon Redshift → query Parquet via external tables

typical pipeline: CSV/JSON (raw) → Glue → Parquet → S3 → Query (fast)

*Delta Lake (Format): An open packet format which is built on the top of packet format and this acts as a transactional layer as well
Delta Lake is: A smart layer on top of Parquet files that adds reliability, versioning, and database-like features.

versioning: every change creates a new version of the table (Time Travel)

What Delta Lake Adds (Super Important)
1. ACID Transactions
Safe writes
No partial/corrupt data

2. Upserts (MERGE)
MERGE INTO table ...
Just like SCD Type 1/2 pipelines

3. Schema Enforcement
Prevents bad data

4. Time Travel
Query old versions

🧠 Real Pipeline Example
Raw Data → Parquet → Delta Lake → Query (fast + reliable)

Used with:Databricks, Spark

🔥 Why Delta Lake Exists

Because Parquet alone is not enough:

Problem in Parquet	        Delta Lake Fix
No versioning	            ✅ Version history
No transactions	            ✅ ACID
Hard updates	            ✅ MERGE support
